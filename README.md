# dsk-keepercmd-demo

End-to-end demo: **keeperCMD migration → dsk import-from-keepercmd → dsk plan**

This repo shows the complete integration story between
[keeperCMD](https://github.com/jlima8900/keeperCMD) (Joao's Keeper tenant-migration toolkit)
and [dsk](https://github.com/msawczynk/dsk) (Martin's declarative Keeper SDK).

---

## The story in three steps

```
keeperCMD                       dsk
─────────────────────────────── ──────────────────────────────────
keeper-migrate run-dir/      ─► dsk import-from-keepercmd run-dir/
  ├── audit.log                   └► manifests/
  ├── manifest.csv                      ├── keeper-shared-folder.SF_ENG_001.yaml
  ├── manifest.csv.sha256               └── keeper-shared-folder.SF_HR_001.yaml
  ├── inventory.json                                   │
  └── SHA256SUMS.txt                  dsk plan ────────┘
                                        → 0 changes (clean)
```

**Step 1 — Migration:** keeperCMD runs `keeper-migrate` against a live (or simulated) tenant,
writes a run-dir with exported records, a manifest CSV, and integrity sidecar files.

**Step 2 — Import:** `dsk import-from-keepercmd run-dir/` consumes the run-dir, verifies the
audit chain and SHA256SUMS, detects any suspect ownership markers, and emits DSK YAML manifests.

**Step 3 — Plan:** `dsk plan` compares the DSK manifests against the live Keeper state.
On first import of freshly migrated records, plan should show zero drift (records are now
adopted declaratively).

```bash
# Install prerequisites
```

## keeperCMD perspective — producing the run-dir

**What keeperCMD is**: a Commander plugin (`keeper_tenant_migrate/`) that extracts an
entire enterprise tenant — nodes, roles, teams, users, records, shared folders,
attachments, enforcement policies — and replays it onto a target tenant with
verify, resume, undo, and audit semantics. 41 subcommands, 2200+ unit tests.
Repo: [jlima8900/keeperCMD](https://github.com/jlima8900/keeperCMD).

**When run-dirs appear**: every mutating `keeper-migrate` invocation writes to a
caller-supplied `--run-dir`. The directory accumulates artifacts across
subcommands within a single migration session — re-running with `--resume`
appends rather than overwrites. The `auto-migrate` orchestrator chains the
canonical subcommand sequence (capture → structure → users → shares →
records-export → records-import → records-references-rewrite → verify) into
one run-dir for an end-to-end migration.

**Subcommand → artifact map** (the four files DSK consumes):

| Artifact | Produced by | Phase | Mutability |
|---|---|---|---|
| `inventory.json` | `live-inventory` (or `inventory --offline-snapshot`) | pre-flight | read-only after capture |
| `audit.log` | every mutating subcommand | continuous | append-only, hash-chained |
| `manifest.csv` | `records-export` then `records-import` | mid-run | append-only with sidecar |
| `manifest.csv.sha256` | `records-import` (write-time sidecar) | mid-run | one per manifest write |
| `SHA256SUMS.txt` | `verify` and `records-export` | post-import | regenerated per pass |

**OUTPUT_CONTRACT.md v1.1**: keeperCMD ships an explicit
[output-contract spec](https://github.com/jlima8900/keeperCMD/blob/master/OUTPUT_CONTRACT.md)
defining the on-disk shape of every artifact above. v1.0 → v1.1 was purely
additive (added the `manifest.csv.sha256` sidecar). Any cross-product consumer
(DSK is the first) should pin to a contract version and treat unknown fields as
forward-compatible.

**Real-tenant invocation (illustrative)**:

```bash
# from keeperCMD's RUNBOOK.md — abridged
keeper-migrate --config /path/to/source.json \
    live-inventory --run-dir /tmp/migration-001
keeper-migrate --config /path/to/target.json \
    auto-migrate --run-dir /tmp/migration-001 \
                 --inventory /tmp/migration-001/inventory.json \
                 --source-read-only
```

This produces the run-dir layout DSK consumes. The full operator workflow lives
in [`keeper_tenant_migrate/RUNBOOK.md`](https://github.com/jlima8900/keeperCMD/blob/master/keeper_tenant_migrate/RUNBOOK.md)
and the architecture organigram in `DEPENDENCY_SCHEMATIC.md`.

**Fixture provenance**: the `run-dir/` checked into this repo is currently
synthetic (3-record). The real anonymized fixture (sourced from rehearsal-16,
keeperCMD's v1.7 marquee live validation pass) will replace it once the
anonymization pipeline runs on the c3por2d2 box.

**Out-of-scope for the demo** (keeperCMD does these but the demo skips them):
attachment streaming, multi-stage resume across crashes, role/enforcement
migration, share-folder permission carry-over, MSP/MC routing. Each is its own
subcommand in keeperCMD; DSK currently focuses on records adoption.

## Security properties demonstrated

| Property | Mechanism |
|---|---|
| Audit chain integrity | `audit.log` SHA256-chained events (every line carries `prev_hash` + `signature`); `AuditChainCorrupt` → hard abort |
| Manifest tamper detection | `manifest.csv.sha256` sidecar check before CSV parse (OUTPUT_CONTRACT v1.1) |
| Record export integrity | `SHA256SUMS.txt` digest verify before record consume |
| Suspect ownership markers | `keeper_declarative_manager` field cross-checked vs `manifest.csv` |
| Atomic durability | Audit-log writes use `tempfile` + `fsync` + `os.replace` + dir-fsync, all under POSIX `flock` (post SEC-3 fix 2026-05-08) |
